## 1. 시스템 개요

### 1.1 이 시스템은?

AI 에이전트를 위한 **로컬 지식기반 메모리** 시스템입니다. Google Gemini의 범용 시맨틱 지식 그래프 연구를 바탕으로, 일상생활부터 코딩, 법무까지 다양한 영역에서 사용할 수 있는 **영속적 지식 누적 시스템**을 제공합니다.

**핵심 철학** (Andrej Karpathy):
> *"Obsidian은 IDE, 언어모델은 프로그래머, 위키는 코드베이스다."*

### 1.2 주요 특징

| 특징 | 설명 |
|------|------|
| **제로 API 비용** | Tree-sitter AST + 로컬 SLM으로 외부 API 의존 없음 |
| **지식 누적** | 매번 처음부터 관계를 탐색하지 않고 지식이 누적됨 |
| **시계열 추적** | 엔티티의 변경 이력을 버전별로 관리 |
| **세션 스코프** | 프로젝트/토픽/세션 계층적 범위 관리 |
| **옵시디언 경험** | Joplin 플러그인으로 [[wiki-links]] 지원 |
| **프로덕션 수준** | 465개 테스트, mypy 0 에러, 81%+ 커버리지 |

## 3. CLI 사용법

### 3.1 기본 명령어

| 명령어 | 설명 |
|--------|------|
| `mnemosyne --version` | 버전 확인 |
| `mnemosyne query --stats` | 그래프 통계 |
| `mnemosyne query --query "search:용어"` | 엔티티 검색 |
| `mnemosyne extract <경로>` | 파일/디렉토리에서 엔티티 추출 |
| `mnemosyne add <대상>` | 파일, 디렉토리, URL, `--text` 텍스트를 그래프에 인제스트 |
| `mnemosyne update <경로>` | 변경된 파일을 그래프와 LLM Wiki에 증분 반영 |
| `mnemosyne wiki <하위명령>` | Markdown LLM Wiki 점검 및 관리 |

### `mnemosyne add` 옵션

| 옵션 | 기본값 | 설명 |
|------|--------|------|
| `--text TEXT` | — | 파일/URL 대신 인라인 텍스트 인제스트 |
| `--domain {coding,daily,legal}` | `daily` | 인제스트 도메인 |
| `--scope-id ID` | — | 인제스트된 엔티티에 스코프 ID 부여 |
| `--source-channel CHANNEL` | `cli` | 인제스트 소스 채널 태그 |
| `--dry-run` | 꺼짐 | 그래프에 쓰지 않고 추출 결과만 미리보기 |
| `--wiki-root PATH` | `~/mnemosyne/wiki` | LLM Wiki 루트 디렉토리 |
| `--no-wiki` | 꺼짐 | 지식 그래프만 갱신하고 wiki 페이지 생략 |
| `--wiki-excerpts` | 꺼짐 | wiki 페이지에 bounded/redacted 원문 발췌 포함 (명시적 opt-in) |

기본 데이터 경로: `~/mnemosyne/raw/`, `~/mnemosyne/wiki/`, `~/mnemosyne/graph/knowledge.db`

### `mnemosyne wiki` 하위 명령어

| 하위 명령어 | 설명 |
|-------------|------|
| `status` | wiki 상태 요약 (페이지 수, stale 수, contradiction 총계 등) |
| `lint` | wiki 링크, 메타데이터, 그래프 drift 점검; `--strict`로 warning 시 실패 처리 |
| `contradictions` | 안정적인 `conflict_id`가 붙은 그래프 기반 conflict review 목록 |
| `resolve <id>` | conflict evidence를 삭제하지 않고 resolution 메타데이터만 갱신 |
| `prune` | stale wiki/graph 정리 계획 미리보기; `--apply-tombstones`로 tombstone 레코드 생성 |
| `semantic-contradictions` | 로컬 오프라인 semantic contradiction 탐지 (opt-in); `--write`로 결과 저장 |
| `rebuild` | 그래프 데이터에서 생성 wiki 섹션 재생성; `--dry-run`으로 미리보기 |
| `doctor` | `status`와 `lint`를 함께 실행 |

`rebuild`, `mnemosyne add`, `mnemosyne update` 등 wiki 쓰기 명령은 wiki root마다 `.mnemosyne-wiki.lock` 파일을 사용합니다 (기본 timeout: 10초).

### 3.2 LLM Wiki + 지식 그래프 인제스트

`mnemosyne add`와 `mnemosyne update`는 구조화된 Knowledge Graph와 함께
Karpathy식 Markdown LLM Wiki를 유지할 수 있습니다.

```bash
# 기본 wiki 위치: ~/mnemosyne/wiki
mnemosyne add ./notes/meeting.md --domain daily --scope-id meeting-demo

# Obsidian/Joplin/git vault 등 원하는 위치에 wiki 쓰기
mnemosyne add ./notes/meeting.md --domain daily --wiki-root ./wiki

# 그래프만 갱신하고 Markdown wiki는 쓰지 않기
mnemosyne add ./notes/meeting.md --domain daily --no-wiki

# 기본값은 안전을 위해 원문 excerpt를 wiki에 복사하지 않음
# 안전한 원본일 때만 bounded/redacted excerpt를 명시적으로 활성화
mnemosyne add ./notes/meeting.md --domain daily --wiki-excerpts

# wiki 상태 점검 / lint / rebuild
mnemosyne wiki status --wiki-root ./wiki --format json
mnemosyne wiki lint --wiki-root ./wiki --format json
mnemosyne wiki rebuild --wiki-root ./wiki --db-path ~/mnemosyne/graph/knowledge.db --dry-run
```

생성/갱신되는 구조:

```text
~/mnemosyne/wiki/
├── index.md              # 내용 중심 카탈로그
├── log.md                # append-only 인제스트 기록
├── sources/<domain>/     # 원본별 요약/출처 페이지
└── entities/<type>/      # 엔티티별 누적 페이지
```

Markdown Wiki는 사람이 읽고 LLM이 유지보수하기 좋은 표현 계층이고,
SQLite + NetworkX는 경로 탐색과 구조화 질의를 위한 지식 그래프 계층입니다.
생성 페이지에는 provenance용 YAML frontmatter가 포함되며, Mnemosyne generated
marker 밖의 수동 노트는 갱신/rebuild 중에도 보존됩니다.

#### Conflict metadata와 contradiction review

인제스트 중 같은 엔티티의 속성값이 충돌하면 Mnemosyne은 기존 값을
덮어쓰지 않고 들어온 값을 `properties["conflicts"]` 아래에 보존합니다.
Wiki는 unresolved conflict metadata를 엔티티 페이지의 generated
`## Potential contradictions` 섹션으로 올려서 사람이 검토할 수 있게 합니다.

- 이 섹션은 deterministic/offline 요약이며, LLM 기반 의미 판단이나
  “거짓” 판정이 아닙니다.
- 충돌 값은 source excerpt와 같은 redaction 정책을 거친 뒤 표시됩니다.
- `mnemosyne wiki status --db-path ... --format json`은 graph DB가 있을 때
  contradiction 총계와 엔티티별 count를 포함합니다.
- `mnemosyne wiki lint --db-path ...`는 unresolved contradiction을 warning으로
  보고합니다. 자동화에서 warning도 실패시키려면 `--strict`를 붙이세요.
- `mnemosyne wiki contradictions --db-path ... --format json`은 stable
  `conflict_id`가 붙은 review item 목록을 보여줍니다.
- `mnemosyne wiki resolve <conflict_id> --resolution accepted_existing`는
  충돌값과 source evidence를 지우지 않고 review metadata만 갱신합니다.
  자동화에서는 먼저 `--dry-run`을 사용하세요.
- Conflict record는 `unresolved`, `accepted_existing`, `accepted_incoming`,
  `superseded`, `ambiguous` resolution 상태를 지원합니다.

#### Optional semantic contradiction discovery

Semantic contradiction discovery는 deterministic conflict metadata와 분리된
opt-in review workflow입니다. 결과는 “진실/거짓” 판정이 아니라 사람이 검토할
candidate입니다.

- `mnemosyne wiki semantic-contradictions --db-path ... --format json`은
  local offline heuristic detector를 실행하지만 review file은 쓰지 않습니다.
- `--write`를 붙이면 `review/semantic-contradictions.*` 아래에 candidate를
  저장합니다.
- 출력은 별도 `mnemosyne.semantic_contradiction_candidates.v1` schema를 쓰며,
  `processing_mode: local-offline`과 remote model 비활성 상태를 명시합니다.
- Candidate evidence에는 source reference, bounded/redacted excerpt,
  confidence, rationale, generated-at metadata, uncertainty wording이 포함됩니다.
  raw source excerpt는 `--include-raw-excerpts`를 명시해야만 포함됩니다.
- `mnemosyne wiki status`와 `mnemosyne wiki lint`는 persisted semantic review
  candidate를 deterministic property conflict와 별도로 보여줍니다. Lint는
  warning만 내며, 자동화에서 실패시키려면 `--strict`를 사용하세요.

#### Stale planning과 tombstone

Mnemosyne은 오래된 wiki/graph lifecycle 정리를 자동 삭제가 아니라 review
작업으로 취급합니다.

- `mnemosyne wiki prune --db-path ... --format json`은 orphan entity/source
  page, missing raw source path 같은 stale candidate의 dry-run plan을 만듭니다.
- Plan에는 candidate ID, path/entity/source, reason, risk label, 가능한 경우
  manual-note preview가 포함됩니다.
- `mnemosyne wiki prune --apply-tombstones`는 `tombstones/` 아래에 Markdown
  tombstone record를 쓰지만 여전히 delete는 0건입니다.
- Tombstone은 recovery metadata와 manual note preview를 담아 사용자가
  의도적으로 archive/cleanup할 수 있게 합니다.
- `mnemosyne wiki status`와 `mnemosyne wiki lint`는 stale count/warning을
  보여주지만 기본적으로 실패시키지 않습니다.

#### Joplin / Obsidian / 동기화 폴더에서 쓰기

에디터가 관리하는 폴더에 쓸 때는 `--wiki-root`를 명시하세요.

```bash
mnemosyne add ./notes/meeting.md --domain daily --wiki-root ~/Notes/mnemosyne-wiki
mnemosyne wiki lint --wiki-root ~/Notes/mnemosyne-wiki
```

사용 계약:

- wiki root는 일반 Markdown 폴더로 열거나 import할 수 있습니다. CI나 기본 사용에 실제 Joplin 실행은 필요 없습니다.
- Joplin에서는 Markdown/folder import 방식으로 사용하세요. 이 워크플로에는 Joplin API token이 필요 없으며, token을 repo나 wiki에 저장하지 마세요.
- Obsidian류 vault에서는 `[[...]]` 링크를 그대로 사용할 수 있게 생성합니다. 단, 에디터마다 `[[path/to/page|label]]` 렌더링이 다를 수 있으므로 broken link 판정은 `mnemosyne wiki lint`를 기준으로 삼으세요.
- `MNEMOSYNE:GENERATED` marker 사이의 텍스트는 rebuild 때 교체됩니다. 사람이 쓰는 노트는 marker 밖, 가능하면 `## Notes` 아래에 작성하세요.
- 원본 raw source와 graph DB가 권위 있는 데이터이고, 에디터 페이지는 사람이 읽는 view + 수동 노트 계층입니다.

#### Wiki writer lock

Wiki를 쓰는 명령은 generated page를 갱신하기 전에 wiki root마다 하나의
`<wiki-root>/.mnemosyne-wiki.lock` 파일을 잡습니다. 이 lock은
`mnemosyne add`, `mnemosyne update`, `mnemosyne wiki rebuild`가 같은 wiki root에
동시에 쓰는 상황을 막습니다.

- 기본 timeout은 10초입니다.
- `mnemosyne wiki rebuild`는 `--lock-timeout <초>`로 대기 시간을 조절할 수 있습니다.
- `mnemosyne wiki status`와 `mnemosyne wiki lint`는 read-only라 기본적으로 lock을 잡지 않습니다.
- 프로세스가 비정상 종료되어 lock 파일이 남으면, 수동 삭제 전에 JSON metadata
  (`pid`, `hostname`, `created_at`, `action`, `wiki_root`)를 확인하세요. Mnemosyne은 stale lock을 자동으로 깨지 않습니다.
- 이 lock은 local filesystem 기준입니다. network/distributed filesystem의 lock semantics는 보장하지 않습니다.

### 3.3 추출 파이프라인

```bash
# 전체 파이프라인 실행
python -m mnemosyne.extraction.pipeline \
  --domain coding \
  --source ~/my-project \
  --format json

# 세션 스코프 및 증분 추출
python -m mnemosyne.extraction.pipeline \
  --domain coding --source ~/my-project \
  --scope-id impl-session --source-channel vscode \
  --incremental

# 시맨틱 추출 제외 (결정적 파싱만)
python -m mnemosyne.extraction.pipeline \
  --domain coding --source ~/my-project --no-semantic
```

### 3.4 결정적 추출 (제로 LLM)

```bash
# 단일 파일
python -m mnemosyne.extraction.deterministic.code_parser ~/project/main.py

# 디렉토리 전체
python -m mnemosyne.extraction.deterministic.code_parser ~/project --format wiki

# 세션 스코프 포함
python -m mnemosyne.extraction.deterministic.code_parser ~/project \
  --scope-id my-session --source-channel code
```

## 4. Python API

### 4.1 엔티티 추출

```python
from mnemosyne.extraction.deterministic.code_parser import TreeSitterExtractor
from pathlib import Path

extractor = TreeSitterExtractor()

# 단일 파일 추출
result = extractor.extract_file_full(Path("main.py"))
print(f"엔티티: {len(result.entities)}개")
print(f"임포트: {len(result.imports)}개")
print(f"호출: {len(result.calls)}개")

# 디렉토리 추출
entities = extractor.extract_directory(
    Path("src/"),
    scope_id="my-session"
)

# 위키 포맷 변환
print(extractor.to_wiki_format(result.entities, result.imports, result.calls))
```

### 4.2 시맨틱 추출

```python
from mnemosyne.extraction.semantic.slm_extractor import SemanticExtractor

# 컨텍스트 매니저로 자동 정리
with SemanticExtractor() as extractor:
    result = extractor.extract(
        "Google Inc의 John Smith",
        ["PERSON", "ORGANIZATION"],
        scope_id="session-1",
        source_channel="cli",
    )
    # result = {entities: [...], relations: [...], token_cost: N}
```

### 4.3 지식 그래프 쿼리

```python
from mnemosyne.graph.knowledge_graph import KnowledgeGraph

kg = KnowledgeGraph()

# 통계
stats = kg.get_stats()

# 검색
results = kg.query("search:authenticate")

# 엔티티 조회
results = kg.query("entity:function[parse_config]")

# 관계 조회
results = kg.query("relation:calls")

# 최단 경로
results = kg.query("path:get_user,authenticate")

# 세션 범위 쿼리
results = kg.query("entity:function[*]@session:impl-session")
```

### 4.4 스코프 관리

```python
from mnemosyne.graph.scope_manager import ScopeManager

sm = ScopeManager()

# 계층적 스코프 생성
sm.create_scope("project-x", scope_type="project")
sm.create_scope("feature-auth", parent_id="project-x", scope_type="feature")
sm.create_scope("session-1", parent_id="feature-auth", scope_type="session")

# 스코프 트리 조회
children = sm.get_children("project-x")
lineage = sm.get_lineage("session-1")  # session-1 → feature-auth → project-x
```

## 5. 도메인 스키마

### 5.1 일상생활 (Daily Life)

| 엔티티 | 설명 | 주요 속성 |
|--------|------|-----------|
| `task` | 작업/할일 | title, deadline, priority, status |
| `person` | 인물/연락처 | name, role, contact, last_interaction |
| `place` | 장소 | name, type, address, visit_frequency |

## 8. 문제 해결

| 문제 | 해결책 |
|------|--------|
| `ModuleNotFoundError: spacy` | `python -m spacy download en_core_web_sm` |
| GLiNER 로드 실패 | `pip install gliner` 후 재시도 |
| Tree-sitter 문법 없음 | `pip install tree-sitter-python` 등 언어별 패키지 설치 |
| Joplin 플러그인 로드 안됨 | Joplin 2.14 이상 확인 |
| 그래프 쿼리 결과 없음 | `search:`로 퍼지 검색 시도 |
| `mypy` 에러 발생 | `pip install -e ".[dev]"` 후 `mypy mnemosyne/` 실행 |
