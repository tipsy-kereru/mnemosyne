## 1. 시스템 개요

### 1.1这是什么系统？

이 시스템은 **AI 에이전트를 위한 로컬 지식기반 메모리**입니다. Google Gemini의 범용 시맨틱 지식 그래프 연구를 바탕으로, 일상생활부터 코딩, 법무까지 다양한 영역에서 사용할 수 있는 **영속적 지식 누적 시스템**을 제공합니다.

**핵심 철학** (Andrej Karpathy):
> *"Obsidian은 IDE, 언어모델은 프로그래머, 위키는 코드베이스다."*

### 1.2 주요 특징

| 특징 | 설명 |
|------|------|
| **제로 API 비용** | 결정적 파싱 + 로컬 SLM으로 외부 API 의존 없음 |
| **지식 누적** | 매번 처음부터 관계를 탐색하지 않고 지식이 누적됨 |
| **시계열 추적** | 엔티티의 변경 이력을 버전별로 관리 |
| **도메인 확장** | 일상/코딩/법무 스키마를 declarative하게 정의 |
| **옵시디언 경험** | Joplin 플러그인으로 [[wiki-links]] 지원 |

### 1.3 아키텍처


## 3. 도메인 스키마

### 3.1 일상생활 (Daily Life)

**대상**: 일정, 연락처, 습관, 선호도, 메모

| 엔티티 | 설명 | 주요 속성 |
|--------|------|-----------|
| `task` | 작업/할일 | title, deadline, priority, status, recurring |
| `person` | 인물/연락처 | name, role, contact, last_interaction |
| `place` | 장소 | name, type, address, visit_frequency |
| `event` | 일정/약속 | title, datetime, duration, participants |
| `habit` | 습관 | name, frequency, time, streak |
| `preference` | 선호도 | category, value, confidence |
| `note` | 일반 메모 | title, content, tags |

**관계 예시**:
```
task.assigned_to → person
task.located_at → place
person.interacts_with → person
event.followed_by → event
```

### 3.2 코딩 (Coding)

**대상**: 소스코드, API, 버그, 기능요청, 테스트, 의존성

| 엔티티 | 설명 | 주요 속성 |
|--------|------|-----------|
| `function` | 함수/메서드 | name, signature, language, parameters, return_type, complexity |
| `class` | 클래스/구조체 | name, methods, attributes, inherits |
| `module` | 모듈/패키지 | name, type, exports, imports |
| `api` | API 엔드포인트 | name, type, endpoint, method, auth_required |
| `bug` | 버그 리포트 | id, severity, status, affected_functions |
| `feature` | 기능 요청 | id, status, implements, priority |
| `test` | 테스트 케이스 | name, type, covers, status |
| `dependency` | 외부 의존성 | name, version, vulnerabilities |

**관계 예시**:
```
function.imports → module
function.calls → function
function.tested_by → test
bug.affected → function
feature.implements → api
```

### 3.3 법무 (Legal)

**대상**: 법령, 조항, 판례, 계약, 당사자

| 엔티티 | 설명 | 주요 속성 |
|--------|------|-----------|
| `statute` | 법률/규정 | name, jurisdiction, code, effective_date |
| `clause` | 조항 | number, title, content, obligation_type |
| `case` | 판례 | citation, court, date, holding, reasoning |
| `party` | 당사자 | name, type, role, counsel |
| `obligation` | 의무/요구 | description, obligated_party, deadline |
| `deadline` | 법적 기한 | date, type, consequence |
| `contract` | 계약 | title, parties, effective_date, status |

**관계 예시**:
```
clause.derived_from → statute
case.applies → statute
obligation.arises_from → clause
contract.between → party
party.against → party
```

## 4. 사용법

### 4.1 추출 파이프라인

#### 코드 엔티티 추출 (제로 LLM)

```bash
# 단일 파일
python -m core.extraction.deterministic.code_parser ~/project/main.py

# 디렉토리 전체
python -m core.extraction.deterministic.code_parser ~/project --format wiki

# 출력 형식
python -m core.extraction.deterministic.code_parser ~/project --format json  # JSON
python -m core.extraction.deterministic.code_parser ~/project --format wiki  # Markdown
```

**지원 언어**: Python, JavaScript, TypeScript, Go, Rust, Java

#### 시맨틱 추출 (로컬 SLM)

```bash
# GLiNER2 + REBEL 사용
python -m core.extraction.semantic.slm_extractor \
  --text "John Smith works at Google in San Francisco" \
  --entities PERSON ORGANIZATION LOCATION

# 파일에서
python -m core.extraction.semantic.slm_extractor \
  --file ~/documents/meeting-notes.txt \
  --entities PERSON ORGANIZATION DATE
```

### 4.2 지식 그래프 쿼리

#### Python API

```python
from core.graph.knowledge_graph import KnowledgeGraph

kg = KnowledgeGraph()

# 엔티티 조회
result = kg.query("entity:function[parse_config]")
result = kg.query("entity:task[.*]")  # 정규식

# 관계 조회
result = kg.query("relation:calls")
result = kg.query("relation:calls(function_id_1, function_id_2)")

# 경로 탐색
result = kg.query("path:get_user,authenticate")
# → 두 함수 간의 최단 경로 반환

# 검색
result = kg.query("search:authentication")

# 통계
result = kg.get_stats()
# → entities, relations, by_type, density 등
```

#### CLI

```bash
# 엔티티 조회
python -m core.graph.knowledge_graph --query "entity:function[.*]"

# 관계 조회
python -m core.graph.knowledge_graph --query "relation:calls"

# 경로 탐색
python -m core.graph.knowledge_graph --query "path:func_a,func_b"

# 검색
python -m core.graph.knowledge_graph --query "search:parser"

# 그래프 통계
python -m core.graph.knowledge_graph --stats
```

### 4.3 위키 링크 문법

Joplin 노트나 일반 Markdown에서 사용:

| 문법 | 의미 | 예시 |
|------|------|------|
| `[[노트명]]` | Joplin 노트 링크 | `[[프로젝트 계획]]` |
| `[[entity:타입:이름]]` | 지식 그래프 엔티티 | `[[entity:function:authenticate]]` |
| `[[entity:person:John]]` | 인물 엔티티 | - |
| `[[entity:task:보고서 작성]]` | 작업 엔티티 | - |
| `[[graph:쿼리]]` | 그래프 쿼리 결과 | `[[graph:search:보안]]` |

**링크 예시**:
```markdown
# 프로젝트 계획

이 프로젝트의 핵심 기능은 [[entity:function:authenticate]] 인증 함수를 사용합니다.

관련 버그: [[entity:bug:AUTH-001]]

法律参考: [[entity:statute:정보통신망법 제23조]]
```

## 5. Joplin 플러그인 기능

### 5.1 플러그인 설치 후 기능

| 기능 | 단축키 | 설명 |
|------|--------|------|
| 검색 | `Cmd+Shift+K` | 지식 그래프 검색 대화상자 |
| 링크 삽입 | `Cmd+L` | 위키 링크 삽입 |
| 그래프 보기 | `Cmd+Shift+G` | 지식 그래프 시각화 |
| 엔티티 추출 | - | 현재 노트에서 엔티티 추출 |

### 5.2 위키 링크 렌더링

플러그인 설치 시:
- `[[note-link]]` → 회색 파선 밑줄
- `[[entity:type:name]]` → 파란색 실선
- `[[graph:query]]` → 주황색 점선

### 5.3 자동 엔티티 추출

노트 저장 시 자동으로 엔티티 추출:
- 태스크 패턴 인식 (`task:`, `TODO:` 등)
- 인물 이름 인식
- 코드 참조 인식

## 7. 기술적 세부사항

### 7.1 추출 파이프라인 비교

| 계층 | 기술 | 도메인 | 하드웨어 | API 비용 |
|------|------|--------|----------|----------|
| **결정적 파싱** | Tree-sitter, SpaCy | 코드, 설정파일 | CPU만 | $0 |
| **로컬 시맨틱** | GLiNER2, REBEL | 비정형 텍스트 | CPU/small GPU | $0 |
| **고급 합성** | Llama-3-8B / GPT-4o | 복잡한 쿼리 | GPU/Cloud | 선택 |

### 7.2 로컬 SLM 모델

**GLiNER2** (NER):
- CPU 실행 가능 (8GB+ RAM 권장)
- 커스텀 엔티티 타입 지원
- 스키마 기반 자연어 설명으로 엔티티 정의

**REBEL** (관계 추출):
- 주어-동사-목적어 삼항组 추출
- 비정형 텍스트에서 관계 파악

### 7.3 그래프 데이터베이스

**저장소**: SQLite (경량, 로컬)
**분석**: NetworkX (경로 탐색, 중심성 분석)
**버전 관리**: entity_history 테이블로 시계열 추적

## 8. 문제 해결

### 8.1 일반 문제

| 문제 | 해결책 |
|------|--------|
| SpaCy 모델 없음 | `python -m spacy download en_core_web_sm` |
| GLiNER 로드 실패 | `pip install gliner` 후 재시도 |
| Joplin 플러그인 로드 안됨 | Joplin 버전 2.14 이상 확인 |
| 그래프 쿼리 결과 없음 | `search:`로 fuzzy 검색 시도 |

### 8.2 성능 최적화

- 코드 추출: Tree-sitter 사용 (GPU 불필요)
- NLP 추출: GPU 있으면 GLiNER2 속도 향상
- 대용량: 배치 처리로 분할 실행

## 10. 예제 워크플로우

### 10.1 새로운 코딩 프로젝트 추가

```bash
# 1. 원본 코드 복사
cp -r ~/Projects/myapp ~/agent-memory/core/raw/coding/myapp

# 2. 추출 실행
python -m core.extraction.deterministic.code_parser \
  ~/agent-memory/core/raw/coding/myapp --format wiki

# 3. wiki 레이어 확인
cat ~/agent-memory/core/wiki/coding/myapp_entities.md

# 4. 그래프에 적재 (API 사용)
python -c "
from core.graph.knowledge_graph import KnowledgeGraph
from core.extraction.deterministic.code_parser import TreeSitterExtractor
from pathlib import Path

kg = KnowledgeGraph()
extractor = TreeSitterExtractor()
entities = extractor.extract_directory(Path('~/agent-memory/core/raw/coding/myapp'))

for e in entities:
    kg.add_entity(Entity(
        id=f\"function:{e.name}\",
        type='function',
        name=e.name,
        properties=e.properties
    ))
"
```
